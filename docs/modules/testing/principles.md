# Testing 模块原理文档（Phase 4+）

## 1. 统一测试入口
- 将测试集中到 `internal/test`，减少测试代码分散在业务目录造成的维护成本。
- 通过 `test.go`（测试逻辑）+ `test_test.go`（入口函数）实现结构化组织。

## 2. 分层隔离与可重复执行
- HTTP 层通过 `VoteClient` mock 和内存实现隔离外部依赖。
- gRPC/Redis 层通过 `miniredis` 提供内存 Redis，保证单元测试在本地与 CI 中可重复执行。

## 3. 行为优先与错误语义稳定
- 断言重点放在外部行为：HTTP 状态码、JSON 响应、gRPC 错误码、结果一致性。
- 显式覆盖 gRPC `InvalidArgument -> HTTP 400` 映射，避免网关语义回归。

## 4. 并发一致性验证
- 通过并发投票测试验证“最终计数 == 总请求数”这一核心不变量。
- HTTP 与 gRPC 两层都加入并发测试，确保链路和业务核心在并发下均保持一致。
- 额外加入 Web 入口全链路集成并发测试，验证请求经过 HTTP->gRPC->Redis 的系统级一致性。

## 5. 可观测性导向测试
- 在测试中加入 `t.Log` 关键过程输出，便于快速定位失败原因。
- 保留开始、关键结果、最终断言日志，提升多人协作时的可读性与排障效率。
